Fortgeschrittene SOQL

Jetzt, da wir wissen, wie man grundlegende SOQL-Abfragen schreibt, müssen wir verstehen, wie man mit SOQL durch Master-Detail-Beziehungen navigiert. Dies ist besonders wichtig, da Sie in realen Szenarien, wenn Sie Daten übergeben, möglicherweise Informationen von mehreren Objekten benötigen, um Daten zu verstehen. Ein zufälliger Kontakt ist beispielsweise für den Endbenutzer nicht viel sinnvoll, es sei denn, sein zugehöriges Konto wird ebenfalls angezeigt.

Standardobjekte

Kind zu Elternteil:

[SELECT ID, LastName, Account.Name FROM Contact WHERE LastName = 'Test'];

Dadurch wird ein Konto zurückgegeben

Elternteil zu Kind:

[SELECT ID, Name, (SELECT ID, LastName FROM Contacts) FROM Account WHERE Name = 'GenePoint'];

Dadurch werden alle Kontakte zurückgegeben, die sich auf dieses Konto beziehen. Beachten Sie auch, dass wir anstelle von Kontakt Kontakte verwendet haben, da jede Unterabfrage mit ihrem Pluralnamen aufgerufen werden muss.

Beziehungsabfragen können ein wenig schwierig zu speichern und darauf zuzugreifen sein. Bringen wir unsere Eltern zum untergeordneten Beispiel, in dem wir alle Kontakte zurückgeben, die sich auf das GenePoint-Konto beziehen, es in einer Kontoliste speichern und in Protokollen ausgeben.

List<Account> finalList = [SELECT ID, Name, (SELECT ID, LastName FROM Contacts) FROM Account WHERE Name = 'GenePoint'];

System.debug(finalList);

Sie werden feststellen, dass Ihr Protokoll nicht die Kontakte zurückgibt, sondern nur Informationen über das Konto:

22:54:11:021 USER_DEBUG [5]|DEBUG|(Account:{Id=0016F00002x0YX7QAM, Name=GenePoint})

Das liegt daran, dass wir die Unterabfrageergebnisse manuell aufrufen müssen, deren Syntax lautet:

Die Syntax für den Aufruf einer Unterabfrage lautet:

listName[0].subQueryObject;
  • listNameIst der Name der Liste.
  • [0], die angibt, dass wir auf das erste Objekt in der Liste zugreifen.
  • subQueryObjectIst der Pluralname des Objekts, das wir in der verschachtelten Abfrage verwendet haben.

Fügen wir dies jetzt unserem Code hinzu

List<Account> finalList = [SELECT ID, Name, (SELECT ID, LastName FROM Contacts) FROM Account WHERE Name = 'GenePoint'];

System.debug(finalList);

System.debug(finalList[0].Contacts);

Jetzt zeigt unser Protokoll die Ergebnisse an (formatiert für leichteres Lesen):

22:54:11:021 USER_DEBUG [5]|DEBUG|(Account:{Id=0016F00002x0YX7QAM, Name=GenePoint})

22:54:11:021 USER_DEBUG [6]|DEBUG|

(Contact:{AccountId=0016F00002x0YX7QAM, Id=0036F00002nVJwYQAW, LastName=Frank},

Contact:{AccountId=0016F00002x0YX7QAM, Id=0036F00003XE2FLQA1, LastName=final})

Benutzerdefinierte Objekte

Navigation through relationships in Custom objects gets a little different from standard objects. This is done so that in future if a new Object is released it doesn't break existing code. When navigation through custom objects, we need to add __r at the end of our object name instead of __c.

Nehmen wir zum besseren Verständnis an, dass Konto und Kontakt benutzerdefinierte Objekte sind.

Kind zu Elternteil:

[SELECT ID, LastName__c, Account__r.Name__c FROM Contact WHERE LastName = 'Test'];

We call LastName with __c because it's a custom field, Account with __r because it's a custom object and it's Name field using __c because it's a custom field.

Elternteil zu Kind:

[SELECT Name__c, (SELECT ID, LastName FROM Contact__r) FROM Account__c WHERE Name__c = 'GenePoint'];

Um diese Daten im Protokoll auszugeben, folgen wir der gleichen Syntax wie bei sObjects.

List<Account__c> finalList = [SELECT Name__c, (SELECT ID, LastName FROM Contact__r) FROM Account__c WHERE Name__c = 'GenePoint'];

System.debug(finalList);

System.debug(finalList[0].Contact__r);

In a parent to child relationship, we call our root (or outer query) using the __c as we would normally, but our subquery uses __r because it's a custom object and we use __r instead of its plural name to future-proof our code. Note that you cannot call a custom object using it's plural name so there are no options here.

Also notice how in Line 3 we use Contact__r instead of Contact__c. For easier understanding, this is because our data type is of Contact__r and not Contact__c.

Zusammenfassung

  • SOQL kann verwendet werden, um durch Beziehungen zu navigieren.
  • Benutzerdefinierte Objekte verwenden __r, während Standardobjekte ihren Pluralnamen verwenden, wenn sie durch Beziehungen zwischen übergeordneten und untergeordneten Beziehungen navigieren.
