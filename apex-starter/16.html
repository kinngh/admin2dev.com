SOQL avanzado

SOQL avanzado

Ahora que sabemos cómo escribir consultas básicas de SOQL, necesitamos entender cómo navegar por las relaciones maestro-detalle usando SOQL. Esto es particularmente importante porque en escenarios del mundo real cuando pasa datos, puede necesitar información de varios objetos para dar sentido a los datos. Por ejemplo, un contacto aleatorio no tendrá mucho sentido para el usuario final a menos que también se muestre su cuenta relacionada.

Objetos estándar

Hijo a padre:

[SELECT ID, LastName, Account.Name FROM Contact WHERE LastName = 'Test'];

Esto devolverá una cuenta

Padre a hijo:

[SELECT ID, Name, (SELECT ID, LastName FROM Contacts) FROM Account WHERE Name = 'GenePoint'];

Esto devolverá todos los contactos relacionados con esa cuenta. También note cómo en lugar de Contacto usamos Contactos, esto se debe a que cada subconsulta debe ser llamada por su nombre plural.

Las consultas de relación pueden ser un poco difíciles de almacenar y acceder. Tomemos nuestro ejemplo de padre a hijo donde estamos devolviendo todos los contactos relacionados con la cuenta GenePoint y lo almacenamos en una lista de cuentas y lo enviemos en registros.

List<Account> finalList = [SELECT ID, Name, (SELECT ID, LastName FROM Contacts) FROM Account WHERE Name = 'GenePoint'];

System.debug(finalList);

Notará que su registro no devuelve los Contactos, sino solo información sobre la Cuenta:

22:54:11:021 USER_DEBUG [5]|DEBUG|(Account:{Id=0016F00002x0YX7QAM, Name=GenePoint})

Esto se debe a que necesitamos llamar manualmente a los resultados de la subconsulta, cuya sintaxis es:

La sintaxis para llamar a una subconsulta es:

listName[0].subQueryObject;
  • listNamees el nombre de la lista.
  • [0]indica que estamos accediendo al primer objeto de la lista.
  • subQueryObjectes el nombre plural del objeto que usamos en la consulta anidada.

Ahora agreguemos esto a nuestro código

List<Account> finalList = [SELECT ID, Name, (SELECT ID, LastName FROM Contacts) FROM Account WHERE Name = 'GenePoint'];

System.debug(finalList);

System.debug(finalList[0].Contacts);

Ahora nuestro registro mostrará los resultados (formateados para facilitar la lectura):

22:54:11:021 USER_DEBUG [5]|DEBUG|(Account:{Id=0016F00002x0YX7QAM, Name=GenePoint})

22:54:11:021 USER_DEBUG [6]|DEBUG|

(Contact:{AccountId=0016F00002x0YX7QAM, Id=0036F00002nVJwYQAW, LastName=Frank},

Contact:{AccountId=0016F00002x0YX7QAM, Id=0036F00003XE2FLQA1, LastName=final})

Objetos personalizados

Navigation through relationships in Custom objects gets a little different from standard objects. This is done so that in future if a new Object is released it doesn't break existing code. When navigation through custom objects, we need to add __r at the end of our object name instead of __c.

Para una comprensión más fácil, supongamos que Cuenta y Contacto son objetos personalizados.

Hijo a padre:

[SELECT ID, LastName__c, Account__r.Name__c FROM Contact WHERE LastName = 'Test'];

We call LastName with __c because it's a custom field, Account with __r because it's a custom object and it's Name field using __c because it's a custom field.

Padre a hijo:

[SELECT Name__c, (SELECT ID, LastName FROM Contact__r) FROM Account__c WHERE Name__c = 'GenePoint'];

Para mostrar estos datos en el registro, seguimos la misma sintaxis que hicimos con sObjects.

List<Account__c> finalList = [SELECT Name__c, (SELECT ID, LastName FROM Contact__r) FROM Account__c WHERE Name__c = 'GenePoint'];

System.debug(finalList);

System.debug(finalList[0].Contact__r);

In a parent to child relationship, we call our root (or outer query) using the __c as we would normally, but our subquery uses __r because it's a custom object and we use __r instead of its plural name to future-proof our code. Note that you cannot call a custom object using it's plural name so there are no options here.

Also notice how in Line 3 we use Contact__r instead of Contact__c. For easier understanding, this is because our data type is of Contact__r and not Contact__c.

Resumen

  • SOQL se puede utilizar para navegar a través de las relaciones.
  • Los objetos personalizados usan __r, mientras que los objetos estándar usan su nombre plural cuando navegan por las relaciones padre a hijo.
