Netzwerk- und REST-API

Sie betreten Ihr örtliches Café, gehen durch die Tür und der Kassierer begrüßt Sie. Sie sind dort ein Stammgast. Sie bestellen Ihre "normale" und die Kasse scheint zu wissen, was Sie damit meinen. Sie machen Sie zu einem supergroßen Frappuccino mit Haselnusstropfen und Schlagsahne. Sie bringen es an Ihren "normalen" Tisch und wenn Sie fertig sind, begrüßen Sie die Kasse und das Personal auf Wiedersehen und machen weiter mit Ihrem Tag.

Abgesehen davon, dass Sie jetzt meine reguläre Kaffeebestellung kennen, wurde der gesamte Prozess durch ein Werkzeug erleichtert, das jeder im Café, einschließlich Ihnen, verstanden hat: Kommunikation. Normalerweise als selbstverständlich angesehen, können Sie diesen gesamten Kurs und alle um Sie herum (oder die meisten von ihnen) aufgrund dieses raffinierten kleinen Werkzeugs verstehen, es könnte die Sprache sein, die Sie sprechen, die schriftliche Nachricht, die Sie weitergeben, oder einfach nur ein Kopfnicken.

Bisher haben wir isoliert mit uns selbst zusammengearbeitet und es ist an der Zeit, dass wir uns in die Außenwelt wagen. Es wird zunächst beängstigend sein, aber mit der Zeit werden Sie mit Ihrer Kommunikation als Entwickler gut umgehen.

Das Tool in unserem Kommunikationsset ist das Verständnis, was eine API ist.

API oder Application Programmable Interface ist eine Middleware für zwei oder mehr Anwendungen im Web, um miteinander zu kommunizieren. Das ist dasselbe wie die Verwendung einer Sprache, um mit einer anderen Person zu sprechen. APIs nehmen eine Eingabe, verarbeiten sie und geben das Ergebnis aus, genauso wie die Frage an die Kasse nach der regulären Bestellung (der Eingabe), der Kasse, die klar versteht, wovon Sie sprechen, und Ihre Endausgabe oder den Kaffee verarbeitet. Es gibt viele Dinge, die in die Erstellung einer guten API einfließen, wie das Schreiben einer großartigen Dokumentation, die klare Anweisungen darüber enthält, welche Art von Eingaben die API nimmt, was sie verarbeitet und was nicht, und wie die Struktur des Endergebnisses aussehen würde. Es gibt viele Dinge, die in die gute API einfließen, aber konzentrieren wir uns im Moment auf das Verständnis und die Verwendung von APIs.

Genau wie echte Sprachen mit vielen Variationen und Zweigen gibt es APIs in zwei Sprachen:

  • REST
  • SEIFE

Im heutigen Kapitel werden wir über die REST-API sprechen.

REST oder REpresentational State Transfer ist aufgrund seiner Struktur und Benutzerfreundlichkeit der häufigere API-Stil, der im Internet verwendet wird. Es gibt viele Konzepte darüber, was REST-API wirklich ist, aber im Moment lassen Sie uns auf einfachem Englisch sprechen, bevor wir uns mit technischen Begriffen befassen.

Lassen Sie uns eine API in Aktion sehen. Dafür brauchen wir zwei Dinge:

  • Etwas, um die REST-API-Aufrufe durchzuführen und ihre Ausgabe anzuzeigen.
  • Ein Server, der REST-Aufrufe akzeptiert und zurückgibt.

Um unsere REST-API-Abfragen auszuführen, verwenden wir Postman. Gehen Sie zu ihrer Website, registrieren Sie sich für ein Konto und laden Sie das Programm für Ihre Plattform herunter.

Um die API in Aktion zu sehen, werden wir Open Weather Map verwenden, um das aktuelle Wetter unseres Standorts oder einer Stadt zu erhalten. Gehen Sie auf ihre Website und melden Sie sich für ein Konto an.

Now in Open Weather Map, head over to the API Section and you will be greeted with a page that shows you "Current and weather forecast weather data collection". We will be using the Current Weather Data collection. Open the API Doc and try not to panic.

Zuerst brauchen wir die API-URL. Im Abschnitt Nach Stadtname werden Sie mit 3 URLs begrüßt:

  • api.openweathermap.org/data/2.5/weather?q={city name}&appid={your api key}

  • api.openweathermap.org/data/2.5/weather?q={city name},{state code}&appid={your api key}

  • api.openweathermap.org/data/2.5/weather?q={city name},{state code},{country code}&appid={your api key}

Für eine REST-API erhalten wir normalerweise zuerst die URL. Wir werden die erste URL verwenden. Kopieren wir das und öffnen wir Briefträger.

Nachdem Sie sich bei Postman angemeldet haben, werden Sie mit dem Begrüßungsbildschirm begrüßt. (Sie können auf Einstellungen > Designs klicken und in den Dunkelmodus wechseln):

Klicken Sie auf Neu > Anfrage und füllen Sie den Namen und die Beschreibung der Anfrage aus.

Dies ist für Ihre Verwendung und Identifizierung, damit die Werte hier das Endergebnis nicht beeinträchtigen. Ich nenne es Wetterdaten und erstelle eine neue Sammlung namens Open Stuff. Sie werden jetzt mit diesem Bildschirm begrüßt:

Im Abschnitt Anforderungs-URL stellen wir zunächst sicher, dass die Anfrage auf GET gesetzt ist (wir sprechen weiter unten in diesem Kapitel darüber) und kopieren die erste URL aus Open Weather Data

api.openweathermap.org/data/2.5/weather?q={city name}&appid={your api key}

Sie werden feststellen, dass Postman die Parameter automatisch aktualisiert. Die URL einer REST-API nimmt Parameter als Eingabezeichenfolgen auf. Wenn dies eine Spitzenmethode wäre, würde sie etwa so aussehen:

public static String weather(String q, String appId){
	String weather = getWeather(q);
	return weather;
}

For the q or city name parameter, write the city you want to get weather for. I'll go with Birmingham (UK)

Jetzt müssen wir unseren API-Schlüssel erhalten. Dies entspricht einem Passwort und muss geheim gehalten werden, da mit diesem Schlüssel Ihre Limits berechnet werden. Nehmen wir an, wenn Ihr API-Schlüssel durchsickern sollte, kann nicht nur jeder jetzt auf Daten hinter der API zugreifen, alle gestellten Anfragen werden jetzt auf Ihre Limits angerechnet, auch wenn es jemand anderes getan hat. Lassen Sie es uns also geheim halten. Um Ihren API-Schlüssel zu erhalten, melden Sie sich bei Open Weather Map an und gehen Sie zu Ihren API-Schlüsseln in Ihrem Dashboard. Geben Sie Ihrem API-Schlüssel einen Namen und klicken Sie auf Generieren.

Kopieren Sie Ihren API-Schlüssel (ich setze meinen zurück, damit er nicht existiert, bitte verwenden Sie diesen nicht) und aktualisieren Sie ihn in Postman. Jetzt haben wir unsere Stadt und unseren API-Schlüssel bereit, klicken wir auf Senden! Wenn der Fehler "Ungültiger API-Schlüssel" angezeigt wird, warten Sie 5 Minuten, da es eine Weile dauert, bis neue API-Schlüssel funktionieren.

Schreiben wir eine JSON-Datei für einen Kontakt mit 3 Verkaufschancen, die mit Account GenePoint verknüpft sind:

{
	"Name": {
		"firstName": "John",
		"lastName": "Smith"
	},
	"Opportinity": [{
			"Name": "Opportunity Name",
			"Close Date": "03022020",
			"Amount": 2200
		},
		{
			"Name": "Opportunity 2 Name",
			"Close Date": "03122020",
			"Amount": 3000
		},
		{
			"Name": "Opportunity 3 Name",
			"Close Date": "12062020",
			"Amount": 800
		}
	],
	"Account": "GenePoint"
}

Lassen Sie uns das aufschlüsseln:

  • First we have the Name object with 2 key value pairs firstName and lastName.
  • We now have an Opportunity Array that holds values for 3 key value pairs Name, Close Date and Amount
  • Wir haben ein Schlüsselwertpaarkonto, das den Wert hältGenePoint

Stellen Sie es sich so vor:

  • Daten werden in Schlüsselwertpaaren gespeichert
  • Wenn es mehr als 1 Schlüsselwertpaar für etwas gibt, definieren wir es als Objekt und speichern es zwischen{ }
  • Wenn es Gruppen von Informationen über ein einzelnes Objekt gibt, aber verschiedene Instanzen definieren, speichern wir sie in einem Array zwischen[ ]

Coming back to our Open Weather Data response, we can now understand the JSON, but in the case of Birmingham, notice how the JSON returned:

    "sys": {
        "type": 1,
        "id": 3436,
        "country": "US",
        "sunrise": 1590921512,
        "sunset": 1590972705
    },

"country": "US"Bedeutet, dass es ein Birmingham in den USA gibt, während wir nach Birmingham in Großbritannien gesucht haben! Die Leute bei Open Weather Data wussten, dass dies ein Problem sein würde, weshalb sie zusätzliche URLs haben, um auf die gleichen Informationen zuzugreifen:

  • api.openweathermap.org/data/2.5/weather?q={city name},{state code}&appid={your api key}

Dadurch können wir den Namen des Bundesstaates und der Stadt definieren und so die Fehlerwahrscheinlichkeit verringern. Dies ist ein Beispiel für ein gutes API-Design, das unserer Kaffeebestellung "Ich werde die Stammgäste haben" entspricht.

Just like Apex has DML statements, REST has it's own version of DML Statements. Remember when we made a GET request in Postman to get information on the URL? That was the equivalent of running a [SELECT all_info FROM Current Weather Data WHERE City = 'Birmingham']. Here are some common DML statements used in REST API calls:

REST-Anruf Apex / Beschreibung

Zusammenfassung

  • APIs sind die Art und Weise, wie Anwendungen über das Internet miteinander interagieren
  • REST und SOAP sind zwei "Sprachen" der API
  • REST wird aufgrund der Benutzerfreundlichkeit und des Verständnisses häufiger verwendet
  • REST hat seine eigenen DML-Anweisungen, genau wie Apex
  • JSON ist ein Format, um Daten organisiert zu speichern
