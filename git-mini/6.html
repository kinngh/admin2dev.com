Zusammenarbeit und Beitrag

Beitrag zu Repos

Nachdem wir nun die Grundlagen der Verwendung von Git kennen, lassen Sie uns mit anderen Entwicklern auf der GitHub-Plattform zusammenarbeiten.

Gabel-Repo

Gehen Sie zum Admin2Dev Learning Markdown Repo und klicken Sie oben rechts auf Fork. Ein Repo zu Forken bedeutet, eine Instanz des Repos für Ihre Verwendung zu erstellen, dies unterscheidet sich vom Klonen eines Repos, da es vom Server auf unser Gerät heruntergeladen wird.

Notice how it says forked from Admin2Dev/learning-markdown right under the repo name on top left. What we have done at this point, is created an instance of the repo for ourselves. By doing this, we can work on this repo however we want without affecting the original repo.

Nehmen wir einige Änderungen vor. Fügen wir in unserer README.MD am Ende eine zusätzliche Zeile hinzu und übergeben sie. Sie können entweder die Schaltfläche "Bearbeiten" auf GitHub verwenden oder das Repository auf Ihrem Computer klonen und hinzufügen/besetzen/pushen.

> May the Salesforce be with you!

Pull Request

Eine Pull-Anforderung von einer Gabelung benachrichtigt den Repo-Eigentümer, sich die von Ihnen vorgenommenen Änderungen anzusehen und sie dem Haupt-Repo hinzuzufügen. In diesem Fall bitte ich Admin2Dev, Besitzer des Learning-Markdown-Repositorys, meine Änderungen in ihrem Repository zusammenzuführen, damit jeder, der das Repo besucht, diese Version sieht.

Klicken wir auf die Pull Request und senden Sie eine Anfrage.

Sie werden feststellen, dass alle vorgenommenen Änderungen aufgelistet sind, zusammen mit einem Bildschirm, der vergleicht, welche Änderungen vorgenommen wurden. In diesem Fall sehen wir eine grüne Linie mit unserem Wortspiel darin, was darauf hinweist, dass wir eine neue Zeile in der README.MD-Datei hinzugefügt haben. Fahren Sie fort und klicken Sie auf Pull Request erstellen und erklären Sie dann, welche Änderungen vorgenommen wurden. Dies ist besonders wichtig, weil Sie dokumentieren müssen, welche Änderungen Sie an der Codebasis vorgenommen haben und warum Sie sie vorgenommen haben.

Wechseln wir nun zur Admin2Dev-Seite (jetzt A2D genannt). A2D erhält eine Benachrichtigung, dass eine neue Pull-Anfrage von kinngh zu ihrem Learning-Markdown-Repository geöffnet wurde.

Beim Öffnen der Benachrichtigung wird A2D auf die Pull Request-Seite weitergeleitet, auf der sie sehen können, welche Änderungen vorgenommen wurden, Commit-Nachrichten, wer die Commits vorgenommen hat, und andere relevante Informationen

Once they confirm the merge, the Pull Request will be closed and the status will be updated to Merged, and the code base was updated with the commit message Merge pull request #1 from kinngh/master. You can now safely delete the original fork you made.

Sie können sich die Git-Repos hier ansehen:

  • Admin2Dev/learning-markdown

Beitragen.md

Viele Open-Source-Projekte werden eine Datei namens CONTRIBUTING.md in ihrem Repository haben, die beschreibt, wie Sie einen Beitrag leisten sollten. Dazu gehört unter anderem sicherzustellen, dass Tests bestanden werden (wie das Schreiben von Testklassen für Apex), der Code wird mit einem bestimmten Linter formatiert, und andere Richtlinien. Sie können einen Blick auf die CONTRIBUTING.md von Alamofire/Alamofire repo werfen, in der beschrieben wird, wie man Fragen stellt, Probleme meldet, Sicherheitsprobleme meldet, grundlegende Verhaltenskodex usw.

Es empfiehlt sich, Zeigern in der Datei CONTRIBUTING.md eines Repos zu folgen, bevor Sie Probleme oder Pull-Requests öffnen.

Lizenzen

Nicht jedes Stück Code auf GitHub ist für die kommerzielle Nutzung oder sogar Open Source für diese Angelegenheit verfügbar. Viele Projekte veröffentlichen ihre Codebasis aus Gründen der Transparenz auf GitHub, weshalb es immer eine gute Idee ist, sich die Datei LICENSE.MD eines Repos anzusehen und ihr tatsächlich zu folgen. GitHub leistet großartige Arbeit bei der Erkennung von Lizenzen und der schnellen Zusammenfassung der Verwendung des Repos, wie dieses von plausible/analytics
